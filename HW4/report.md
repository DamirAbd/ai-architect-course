# Домащнее задание 4

## Постановка задания

В вашем проекте возникла дилемма. Нужно выбрать, где хостить LLM: использовать проприетарную облачную модель (SaaS, напр. GPT-5) или развернуть Open-Source модель (Llama 3) на своих серверах (Self-hosted).

## Решение

###  Сравнение вариантов

| Критерий                          | SaaS LLM                                                                                                                            | Self-hosted LLM                                                                              |
| --------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------- |
| **Стоимость**                     | Низкие затраты на инфораструктуру на входе. Не нужна своя комада сопровождения (devOps). При высокой нагрузке стоимость будет выше. | Высокие стартовые затраты. При большом стабильном объеме запросов может стать выгоднее SaaS. |
| **Безопасность данных / Privacy** | Передача данных внешнему провайдеру. Дополнительные затраты при работе с перс. данными.                                             | Все данные в своем контуре, ниже риск утечки данных.                                         |
| **Качество ответов**              | Качество лучше.                                                                                                                     | Качество ниже. Большие затраты при необходимости повышения качества.                         |
| **Latency**                       | Внешняя зависимость. Задеркжи выше.                                                                                                 | Очень хорошая при локальном размещении. При нехватке GPU latency резко ухудшается.           |
| **Сложность поддержки**           | Низкая: провайдер управляет моделью, обновлениями, масштабированием, отказоустойчивостью.                                           | Высокая: нужно поддерживать инфраструктуру, обновления.                                      |

### Выбор

Использовать SaaS LLM.

Для нашего случая необходима скорость развертывания и минимизация рисков. Низкие затраты на инфраструктуру в начале проекта. Нет необходимости нанимать команду для сопровождения. Высокое качество ответов "из коробки". SaaS LLM обеспечивает лучшее соотношение time-to-market / качество / сложность внедрения.

#### ADR-001: Выбор способа размещения LLM — SaaS (GPT-5) vs Self-hosted (Llama 3)

##### Context and Problem Statement

В рамках разработки AI-функционала (генерация ответов, обработка пользовательских запросов, интеграция с RAG) необходимо определить способ размещения LLM: использовать внешнюю SaaS-модель (например, GPT-5) или развернуть open-source модель (например, Llama 3) на собственной инфраструктуре.

##### Decision Drivers

* Сокращение time-to-market (быстрый вывод в прод)
* Высокое качество ответов без дополнительного обучения модели
* Минимизация инженерной и операционной сложности
* Обеспечение безопасности и приватности данных
* Предсказуемость и контроль затрат
* Снижение latency и стабильность работы

##### Considered Options

* SaaS LLM (GPT-5 или аналог)
* Self-hosted LLM (Llama 3 на собственной инфраструктуре)

##### Decision Outcome

Выбрано: **"SaaS LLM"**. Данное решение обеспечивает наилучшее соотношение скорости внедрения, качества ответов и сложности эксплуатации в текущих условиях проекта.

###### Consequences

* "+": существенно сокращается время внедрения (отсутствует необходимость разворачивания ML-инфраструктуры)
* "+": обеспечивается высокое качество ответов без fine-tuning
* "+": снижается нагрузка на команду (не требуется поддержка GPU, inference и жизненного цикла моделей)
* "-": появляется зависимость от внешнего провайдера (vendor lock-in)
* "-": because возникают риски, связанные с передачей данных за пределы контура компании
* "-": because стоимость может увеличиваться пропорционально росту нагрузки

###### Confirmation

Соответствие подтверждается через:

* Архитектурный review;
* Code review;
* Нагрузочное тестирование;
* Мониторинг затрат;

##### Pros and Cons of the Options

###### SaaS LLM

Использование внешнего LLM-провайдера через API.

* "+": минимальный time-to-market;
* "+": высокое качество ответов “из коробки”;
 *"+": отсутствует необходимость в инфраструктуре и поддержке моделей;
* "+/-": latency зависит от сети и провайдера;
* "-": зависимость от внешнего сервиса (SLA, rate limits);
* "-": ограниченный контроль над моделью;
* "-": потенциальные риски по безопасности данных

###### Self-hosted LLM

Развертывание open-source модели на собственной инфраструктуре.

* "+": полный контроль над данными;
* "+": независимость от внешних провайдеров;
* "+": потенциально ниже стоимость при больших объемах нагрузки;
* "+/-": качество можно улучшить, но требуется затраты на RAG/fine-tuning;
* "-": высокий порог входа (GPU, инфраструктура, команда);
* "-": высокая сложность эксплуатации;
* "-": увеличенный time-to-market;
* "-": риск более низкого качества

##### Доп. информация

* Архитектура должна предусматривать LLM Gateway, чтобы обеспечить возможность смены модели или перехода на self-hosted решение.
* Для снижения рисков безопасности необходимо:
  - минимизировать передачу чувствительных данных,
  - использовать маскирование/обезличинность данных,
  - внедрить аудит и логирование запросов.
* Решение подлежит пересмотру при:
  - росте нагрузки,
  - ужесточении требований по безопасности,
  - появлении более эффективных open-source моделей.
* В перспективе возможен гибридный сценарий: SaaS для сложных задач и self-hosted для типовых запросов.

### Pitch

В текущих условиях SaaS-модель — это рациональный выбор, потому что она дает нам максимальную скорость запуска, лучшее качество ответов и минимальную операционную нагрузку на команду. Вместо того чтобы тратить месяцы на GPU-инфраструктуру, инференс, мониторинг и доведение open-source модели до приемлемого уровня, мы можем сразу вывести ценность в продукт и проверить гипотезу на сильной модели. При этом архитектурно мы не закрываем себе путь к self-hosted сценарию в будущем: если вырастут требования по privacy, контролю данных или экономике на больших объемах, мы сможем перейти на собственный контур позже, уже имея подтвержденный бизнес-кейс.
